# Orders / Market

The **Orders** module (also called **Market**) is a cross-server buy-order system built into OreoEssentials.
Players post requests saying "I want to buy X of item Y at Z price each."
Other players on any server in the network then deliver those items and get paid instantly.

---

## Concept

Unlike a typical auction house (where players sell items), the Orders system works the other way around:

- A **requester** creates a buy order: "I want 64 Diamond Swords, I will pay $10 each."
- The requester's money (escrow) is locked immediately when the order is created.
- Any other player who has the items can **fill** that order — they hand over the items and receive the money.
- The order stays open until it is fully filled or the requester cancels it.
- If an order is partially filled, it stays active until the remaining quantity reaches zero.

This makes it a player-driven economy tool where demand is expressed as open orders and supply is filled on-demand.

## GUI Overview

The Orders module has several GUI screens. Players flow between them naturally.

### 1. Market Browser (`/order`)

The main screen. Shows all currently ACTIVE orders from all players on all servers.

- Each order is displayed as the item being requested, with its lore showing:
  - Who requested it
  - Remaining quantity / total quantity
  - Unit price
  - Currency
  - Remaining escrow (funds still locked)
  - Custom item tags (ItemsAdder / Nexo / Oraxen) if applicable
- **Clicking your own order** takes you to your My Orders screen.
- **Clicking someone else's order** opens the Fill Order screen.
- Supports **pagination** — use the Previous/Next page arrows at the bottom.
- Has two action buttons at the bottom:
  - **Create Order** — starts the order creation flow.
  - **My Orders** — shows your own active orders.
- Updates **live in real time** without closing or reopening — see the Live Updates section.

---

### 2. Create Order Flow (5 steps)

Creating an order is a guided multi-step process:

#### Step 1 — Confirm Item
- The player must hold the item they want to buy in their **main hand**.
- A GUI opens showing the item with a Confirm and a Cancel button.
- The confirmed item becomes the "template" — fillers must deliver an item that matches this exactly.

#### Step 2 — Enter Quantity (chat input)
- The GUI closes and the player is asked to type the quantity in chat.
- Must be a positive integer between 1 and `max_quantity_per_order` (configurable, default 100,000).
- Typing an invalid value keeps the player in this step and shows an error.

#### Step 3 — Select Currency
- A GUI opens showing available currency options.
- If **Vault** is enabled, a gold ingot button appears for standard server money.
- Each registered **custom currency** (OreoEssentials Currency module) gets its own button.
- Which options appear is controlled by `allow_vault` and `allow_custom_currencies` in settings.

#### Step 4 — Enter Unit Price (chat input)
- The GUI closes and the player types the price per single item in chat.
- Must be at or above `min_unit_price` (configurable, default $0.01).
- Both `.` and `,` are accepted as decimal separators.

#### Step 5 — Confirm & Pay
- A summary GUI shows:
  - Item name
  - Quantity
  - Unit price
  - Currency
  - Total escrow (unit price × quantity)
  - Creation fee (if fees are enabled)
  - Total amount that will be charged now
- The player clicks **Confirm & Pay** to submit, or **Cancel** to go back to the browser.
- On confirm, the full escrow (plus creation fee if applicable) is **immediately withdrawn** from the player's balance.
- If the payment fails (insufficient funds, DB error), the order is never created and the player gets an error message. If a DB write fails after withdrawal, the escrow is automatically refunded.

---

### 3. Fill Order Flow (3 steps)

When a player clicks someone else's order in the browser:

#### Step 1 — Order Details
- Shows a detailed GUI with:
  - Who requested the order
  - Item name
  - Remaining / total quantity
  - Unit price and currency
  - How much the filler will earn per item (after fill fee if applicable)
  - The actual item icon (supports ItemsAdder / Nexo textures)
- Two buttons: **Fill This Order** and **Back**.

#### Step 2 — Enter Fill Quantity (chat input)
- The player types how many they want to deliver (between 1 and the current remaining quantity).
- The remaining quantity is re-checked live against the in-memory cache at this point.

#### Step 3 — Confirm Fill
- Summary GUI showing:
  - What item and how many will be taken from inventory
  - How much money will be received (net, after fill fee if applicable)
  - Fill fee amount (only shown if fees are enabled)
- **Confirm** button triggers the actual fill.
- On confirm:
  1. Items are taken from the filler's inventory on the main thread.
  2. An atomic database write updates the order's remaining quantity and escrow.
  3. If the DB write fails (race condition — someone else filled it first), items are returned to the filler.
  4. If it succeeds, the filler is paid. The order creator receives their items (see Item Delivery below).

---

### 4. My Orders

Accessible from the browser or by clicking your own order. Shows all your currently ACTIVE orders.

Each order entry shows:
- Item icon and name
- Remaining / total quantity
- Unit price
- Remaining escrow locked

Clicking an order opens a **Cancel Confirm** screen:
- Shows item, remaining quantity, and the escrow amount that will be refunded.
- **Yes, Cancel & Refund** — cancels the order in the database and refunds the remaining escrow to your balance.
- **No, Keep Order** — goes back to My Orders.

## Item Delivery

When a filler fills an order, the items go to the **order requester**.

- **If the requester is online on the same server** — items are given directly to their inventory. If inventory is full, items are dropped at their feet.
- **If the requester is offline or on another server** — a **Pending Delivery** record is saved to the database. The next time that player joins any server running OreoEssentials, the items are automatically given to them (handled by `OrdersJoinListener`). This prevents item loss for cross-server fills.

---

## Live GUI Updates

The market browser updates itself live without any close/reopen animation.

When any order event happens (new order created, order filled, order cancelled) — on this server or any other server in the network:

1. The event is broadcast via RabbitMQ to all servers.
2. Every server applies the change to its in-memory order cache.
3. All players currently viewing the market browser have their GUI marked as "dirty."
4. On the very next server tick, the GUI is re-rendered in-place — items update seamlessly while the player is watching, with zero flicker.

The refresh is debounced (configurable via `live_refresh_debounce_ticks`, default 10 ticks / 0.5 seconds) to batch rapid events together and avoid spam-refreshing.

## Cross-Server Synchronization

The Orders module uses **RabbitMQ** (via OreoEssentials' packet system) to synchronize orders across all servers in real time. This requires:

- `cross_server.enabled: true` in `orders/settings.yml`
- MongoDB as the storage backend (`storage.type: mongodb`)
- A working RabbitMQ connection configured in the core `config.yml`

### How it works

Every time an order is created, filled, or cancelled on Server A:

1. Server A saves the change atomically to **MongoDB** (shared across all servers).
2. Server A publishes an `OrderSyncPacket` to RabbitMQ.
3. Every other server receives the packet and applies the change to its **in-memory cache**.
4. Each server then triggers a live GUI refresh for any players viewing the market browser.

The packet contains a **revision number** — servers ignore packets with a revision equal to or lower than their local copy (stale-event guard). This prevents out-of-order packet delivery from corrupting the cache.

### What each packet type carries

| Type | Contents |
|---|---|
| `ORDER_CREATED` | Full order data: requester UUID/name, item (Base64), display name, total qty, unit price, escrow, currency, revision |
| `ORDER_UPDATED` | Order ID, new remaining qty, new escrow, new status, revision |
| `ORDER_REMOVED` | Order ID, revision, final status |

### Same-server behavior (echo suppression)

Each server stamps its own `serverId` into every outgoing packet. When a server receives a packet with its own `serverId`, it silently ignores it. This prevents the originating server from double-processing its own events.

### SQLite mode (single server only)

If you use `storage.type: sqlite`, cross-server sync is **not available** even if `cross_server.enabled: true`. SQLite is a local file — other servers cannot access it. A warning is logged at startup if this misconfiguration is detected. In SQLite mode everything works perfectly on a single server.

## Storage

The module supports two storage backends, configured in `orders/settings.yml`:

### MongoDB (recommended for cross-server)

```yaml
orders:
  storage:
    type: mongodb
    mongodb:
      collection: oreo_orders   # collection name (prefix from core config is applied if default)
```

- Uses the MongoDB connection URI from the core `config.yml` (`storage.mongo.uri`).
- Atomic fill operations use an aggregation pipeline with `$cond` to set the order as COMPLETED when `remainingQty` hits zero — no separate update step, no race condition.
- Atomic cancel uses a find-and-update with a status guard.

### SQLite (single-server, no setup required)

```yaml
orders:
  storage:
    type: sqlite
    sqlite:
      file: orders.db   # stored in plugins/OreoEssentials/orders/
```

- Uses a synchronized JDBC transaction with `BEGIN IMMEDIATE` for atomic fills.
- No external dependency — works out of the box.
- Pending deliveries are always stored in SQLite regardless of the main storage type.

## Economy & Currency

The module supports two currency types:

| Type | How to select | Notes |
|---|---|---|
| Vault | Leave currency as null / select gold ingot in the picker | Uses the server's primary economy (Vault) |
| Custom Currency | Select it in the currency picker during order creation | Uses OreoEssentials' CurrencyService |

Control which types are available:

```yaml
orders:
  economy:
    allow_vault: true
    allow_custom_currencies: true
    default_currency: money
```

If `allow_vault: false`, the gold ingot button is hidden from the currency picker. If `allow_custom_currencies: false`, custom currency buttons are hidden.

---

## Fees

Optional transaction fees can be taken from both sides:

```yaml
orders:
  fees:
    enabled: false
    create_fee_percent: 0.0   # charged to the requester on top of escrow
    fill_fee_percent: 0.0     # deducted from the filler's payout
```

- **Creation fee** — taken at order creation time. Shown in the confirm screen before the player agrees. Not refunded if the order is cancelled.
- **Fill fee** — deducted from the filler's gross payout. Shown in the fill confirm screen. The net amount is what the filler actually receives.

Example: `fill_fee_percent: 5.0` means if an order pays $100 gross, the filler receives $95.

---

## Limits

```yaml
orders:
  limits:
    max_active_orders_per_player: 10    # how many open orders one player can have at once
    max_quantity_per_order: 100000      # maximum items per order
    min_unit_price: 0.01                # minimum allowed unit price
```

- Players who exceed `max_active_orders_per_player` see an error and cannot create more orders until some are filled or cancelled.
- `max_quantity_per_order` is validated at creation time and enforced in the chat input step.

---

## Custom Items (ItemsAdder / Nexo / Oraxen)

The orders system fully supports custom item plugins. When an order is created:

- If the item in hand is recognized as an **ItemsAdder**, **Nexo**, or **Oraxen** custom item, its custom ID is stored alongside the standard serialized item data.
- When displaying order icons in any GUI, the plugin first tries to reconstruct the item using the custom ID (so the correct texture and NBT are shown even on servers that didn't originate the order).
- Matching for fill validation also uses the custom item API when available.
- This means custom items survive cross-server sync correctly — a player on Server B can see and fill an order for a custom item created on Server A.

## Configuration Files

All configuration lives in `plugins/OreoEssentials/orders/`.

### `settings.yml` — Main Configuration

```yaml
orders:
  enabled: true
  debug: false           # enables verbose logging for every order event

  storage:
    type: sqlite           # "sqlite" or "mongodb"
    sqlite:
      file: orders.db
    mongodb:
      collection: oreo_orders

  cross_server:
    enabled: false         # requires mongodb + RabbitMQ
    exchange: oreo.orders

  limits:
    max_active_orders_per_player: 10
    max_quantity_per_order: 100000
    min_unit_price: 0.01

  economy:
    allow_vault: true
    allow_custom_currencies: true
    default_currency: money

  ui:
    live_refresh_debounce_ticks: 10   # ticks between receiving an event and refreshing GUIs

  fees:
    enabled: false
    create_fee_percent: 0.0
    fill_fee_percent: 0.0

  lang: en     # language code; loads lang_<code>.yml
```

### `gui.yml` — GUI Appearance

Controls titles, button slots, materials, and names for every GUI screen. Each section corresponds to one GUI:

- `browse` — the main market browser
- `my-orders` — the player's own orders list
- `fill` and `fill.confirm` — the fill flow
- `create.item`, `create.currency`, `create.confirm` — the creation flow steps
- `cancel-confirm` — the cancel confirmation screen

### `lang_en.yml` / `lang_fr.yml` — Language Files

All player-facing messages are in these files. You can create your own `lang_xx.yml` and set `lang: xx` in settings to use it.

## Startup Sequence

On server start, the module initializes in this order:

1. Load `settings.yml`, `gui.yml`, and the language file.
2. Connect to storage (MongoDB or SQLite). If MongoDB fails, falls back to SQLite automatically.
3. Register the `OrdersEventBus` with the server ID.
4. Register `OrdersJoinListener` (handles pending deliveries for returning players).
5. Load all active orders from the database into memory (`OrderService.loadActive()`).
6. After RabbitMQ is connected (done separately in OreoEssentials core init), subscribe to `OrderSyncPacket` events.
7. The module is marked as ready. Log line: `[Orders] Module ready (storage=..., cross-server=...)`.

## Admin Reference

### Reload the module
```
/order reload
```
Requires `oreo.orders.admin`. Reloads all config files, reconnects storage, reloads active orders from the database. Running players' GUIs are not forcibly closed.

### Debug mode
Set `debug: true` in `settings.yml` and run `/order reload`. Every order event (create, fill, cancel, cross-server receive, cache update) will be logged to the console with full detail. Useful for diagnosing cross-server sync or fill issues.

### Checking cross-server status
At startup the console logs:
```
[Orders] MongoDB target: database="<db>" collection="<collection>"
[Orders] Module ready (storage=mongodb, cross-server=true)
[Orders] Cross-server sync subscribed.
```
If you see `storage=sqlite` when you expected `mongodb`, check your MongoDB URI and connection.

If you see:
```
*** MISCONFIGURATION: storage=mongodb in config but running on SQLite! ***
```
It means MongoDB is configured but the connection failed. Cross-server fills will not work until this is fixed.

## Common Questions

**Q: A player created an order on Server A. Can a player on Server B fill it?**
Yes, as long as `cross_server.enabled: true` and MongoDB is configured. The order appears in the browser on every server within the debounce window (~0.5 seconds by default).

**Q: What happens if the order creator is offline when someone fills their order?**
The filled items are saved as a Pending Delivery in the database. When the creator next joins any server, `OrdersJoinListener` detects it and gives them the items automatically.

**Q: Can I use SQLite on a multi-server network?**
No. SQLite is a local file — servers cannot share it. Use MongoDB for cross-server setups.

**Q: Can I have both Vault and custom currencies in the same server?**
Yes. Both can be enabled simultaneously. Players choose which currency to use when creating each individual order.

**Q: What happens if a player tries to fill an order that another player already fully filled a second ago?**
The atomic database update will find `remainingQty` already at 0 and return a `ALREADY_CLOSED` result. The filler's items are returned to their inventory immediately and they see an error message.

**Q: How many items can be in the market at once?**
There is no hard limit on the total number of active orders across all players. Each player is individually limited by `max_active_orders_per_player`.
